# skill: expert-panel — совет гетерогенных моделей вместо одного ответа

> Агент читает это как playbook: любой концептуальный артефакт (правило `agents/*.md`,
> скил, спека, материализация рецепта под язык, архитектурное решение) ты **собираешь и
> судишь СОВЕТОМ из разных моделей, не соло**. Пул и ключ — из окружения, скил не привязан
> к вендору. Дефолтный пул дублирован плейсхолдерами `<МОДЕЛЬ_1..3>` под замену дома.

## Когда применять
Генерация или ревью артефакта, где цена ошибки выше стоимости 3–5 вызовов моделей.
Триггеры: «собери советом», «оцени панелью», «материализуй советом рецепт под язык».
Соло-генерация выдаёт усреднённую по туториалам заготовку: разделы на месте, библиотеки
названы — а дыры системные, и спросишь модель «довольна ли?» → довольна (self-preference).

## Пул (3 разных семейства) и как физически позвать модель
Совет = **3 модели трёх РАЗНЫХ семейств** (не три копии одной: у копий общие слепые пятна
и мягкий self-судья). Дефолт воркшопа: `deepseek/deepseek-v4-flash` · `z-ai/glm-5.2` ·
`moonshotai/kimi-k2.7-code`.

Каждого судью/креативщика зовёшь САМ, отдельным изолированным HTTP-вызовом к шлюзу
(ключ — из окружения `$OPENAI_API_KEY`, В ФАЙЛ НЕ ПИСАТЬ):

```bash
curl -s --max-time 600 "$OPENAI_BASE_URL/chat/completions" \
  -H "Authorization: Bearer $OPENAI_API_KEY" -H "Content-Type: application/json" \
  -d '{"model":"deepseek/deepseek-v4-flash","stream":false,
       "provider":{"order":["baidu","together"],"allow_fallbacks":true},
       "messages":[{"role":"system","content":"<промпт роли>"},
                   {"role":"user","content":"<артефакт + факт-карта + критерии + формат VERDICT=>"}]}'
# ответ модели — в .choices[0].message.content
```

- **Пин провайдера обязателен для скорости.** Шлюз — прокси на OpenRouter; без `provider`
  дефолт-роутинг лотерит и роняет модель на медленного провайдера (10 tok/s). Пин:
  deepseek → `"order":["baidu","together"]`; glm → `"provider":{"sort":"throughput"}`.
- **Роли НЕ прибиты к моделям:** на каждый раунд перемешивай «роль→модель» детерминированной
  перестановкой (по номеру цикла и дню месяца) и пиши назначение в footer вердикта — иначе
  прогон невоспроизводим и не понять, чей вердикт какой моделью получен.
- Пул недобран (нет ключа / лимит) → 1 retry → пометка «пул N/3», не выдавай за полную тройку.

## Роли судей (конвергентная фаза — оценка)
| Роль | Фрейм | Вопрос роли |
|---|---|---|
| Скептик | риски, что сломается | «Почему это НЕ сработает?» |
| Реалист | исполнимость, ресурсы | «Что нужно, чтобы взлетело?» |
| Визионер | масштаб, деградация | «Выдержит ли рост?» |

Каждый судья одевает шапку эксперта предметной области артефакта (хранилище → как DBA,
сеть → как DevOps) и даёт КОНКРЕТНЫЕ советы, не «улучши обработку ошибок».

## Две фазы — не смешивать
1. **Дивергентная (генерация/материализация):** модели независимо предлагают/материализуют;
   оркестратор дедуплицирует, собирает черновик. **Оценивать здесь ЗАПРЕЩЕНО** — ранняя
   критика рубит варианты до того, как дозреют.
2. **Конвергентная (суд):** судьи независимо оценивают готовый черновик.
Прямой стрелки «генератор → судья» нет: артефакт всегда проходит сборку черновика.

## Формат вердикта (жёсткий — иначе голоса не собрать механически)
Каждый судья отвечает строго:
```
<КРИТЕРИЙ>: VERDICT=PASS|FAIL|INCONCLUSIVE — обоснование
… (по строке на каждый критерий)
ОБЩИЙ ВЕРДИКТ: PASS|NEEDS WORK|RED FLAG
СПИСОК ПРАВОК: (при любом не-PASS — все сразу, нумерованным списком)
```
Ответ без строк `VERDICT=` = **падение судьи** (1 повтор → замена другой моделью пула),
а не «попробуем распарсить эссе». Вердикт по обрывку не угадывают.

## 6 критериев
Корректность · Полнота · Реализуемость · Риски · Альтернативы · Соответствие контексту.
Для **материализации под язык** добавь 7-й — **Паритет:** реализуют ли языки один инвариант
ОДИНАКОВО наблюдаемо (иначе под одной спекой поведение разъедется).

## Раунды: R1 → выбор ветки R2 (не гонять второй раунд вслепую)
**R1 — челлендж:** 3 судьи ПАРАЛЛЕЛЬНО и ИЗОЛИРОВАННО (никто не видит чужих ответов —
против anchoring: увидел чужой вердикт → прилипнет). Пиши каждый вердикт сразу в файл.
После R1 выбери ОДНУ из 4 веток:
1. **Все PASS, обоснования РАЗНЫЕ** → R2 не нужен, финальный PASS.
2. **Все PASS, обоснования ПОХОЖИЕ** (подозрительно: ролевое разделение не удержалось) →
   **forced-dissent:** 1 судья ищет контраргумент. Подтвердил PASS → стоп; нашёл FAIL → дебат.
3. **Вердикты РАЗОШЛИСЬ** → **полный дебат:** 3 судьи, каждый теперь ВИДИТ чужие вердикты R1
   и уточняет позицию. Обмен мнениями — только ЗДЕСЬ, после независимого R1.
4. **Единогласный FAIL** → R2 пропустить, сразу правки → **полный новый цикл с R1**
   (никогда не «ещё один R2 поверх старого»).

## Агрегация
- **Majority по каждому критерию:** 2–3 FAIL из 3 → критерий FAIL; 0–1 → PASS. Пиши счёт
  (`Retry FAIL 3:0`, `Таймаут PASS 2:1`). INCONCLUSIVE в подсчёт не идёт (пометка «требует проверки»).
- **Взвешенная сумма провалов:** обычный FAIL = 1.0; усиленный режимом = 1.5; NEEDS WORK
  из деградации 2/3 = 0.5. Пороги: **сумма 0 = PASS · 0<сумма<3.0 = NEEDS WORK · ≥3.0 = RED FLAG**.
  Дробную сумму пиши рядом — иначе прогоны несравнимы, «стало лучше?» превращается во вкусовщину.

## Цикл до PASS большинства
NEEDS WORK/RED FLAG → закрой ВСЕ majority-FAIL правками → **свежий прогон** (правка «на слово»
не считается, проверяется судом). Лимит 3–5 циклов; не сошлось → стоп, остаток в roadmap,
эскалация человеку. Новый раунд = свежие изолированные вызовы с перемешанными ролями;
спрашивать того же судью «теперь доволен?» ЗАПРЕЩЕНО (он вложился и одобрит).

## Анти-галлюцинация — 3 защиты от ложных FAIL (+ калибровка)
1. **INCONCLUSIVE:** судья без web-поиска не имеет права валить внешний факт («такой версии
   нет», «модель устарела») — ставит INCONCLUSIVE. FAIL по факту — только при противоречии
   ВНУТРИ артефакта (число в таблице ≠ число в тексте).
2. **Факт-карта:** у судей инструментов нет, у оркестратора есть. ДО суда извлеки из артефакта
   проверяемые сущности (модели/версии/цены/цитаты/arXiv-ID), проверь поиском, собери
   `fact-card.md` (сущность: есть да/нет/неизвестно · версия · источник-URL) и вложи в промпт
   КАЖДОМУ судье: «есть в карте — существует, не ставь FAIL по несуществованию». 1 раз на цикл.
3. **Grep-проверка «пробела, которого нет»:** FAIL вида «нет X / не описано Y» → оркестратор
   ищет X прямо в тексте артефакта; нашёл → FAIL невалиден (судья не дочитал), в majority не идёт.
**Калибровка:** системно строгий судья (в пуле — kimi) засыпает критику придирками, и majority
не всегда спасает → каждый его FAIL верифицируй grep/поиском ДО зачёта (отделяем шум от находок).

## Деградация и бюджет
- Судья упал (таймаут/ошибка/пустой; **malformed = падение**) → 1 retry → откат на модель
  сессии (кворум 3 сохранён). Лестница: 0 упало = полный · 1 = degraded (majority по 2,
  расхождение 1:1 → NEEDS WORK вес 0.5) · 2 = low-confidence (вердикт одного, пометка) · 3 = abort.
- **≥2 судьи откатились на одну модель → «гетерогенность НЕ достигнута»** (footer не должен
  врать, будто судили три семейства, когда судила почти одна модель).
- Бюджет: потолок 7 вызовов судьи (3 R1 + 1 forced-dissent + 3 дебат), с retry — 9. В retry/
  fallback уходит ≤2 судей (падает третий → строка деградации, не новые попытки). Раунд 3+ нет.
- **Цена полного прогона (замерено на шлюзе):** R1+R2 = 6 вызовов ≈ **$0.05 · ~2 мин · ~35k токенов**; `kimi` ≈ 2/3 стоимости и потолок времени. С forced-dissent/retry (7–9 вызовов) → ≈ $0.08. Дёшево — потому совет и оправдан там, где цена ошибки выше нескольких вызовов.

## Куда что пишется (дисциплина в артефактах, не в памяти оркестратора)
Файлы прогона — в подкаталоге `panel/<slug>/` ТЕКУЩЕГО рабочего каталога (относительный путь, НЕ абсолютный `~/...`: headless-агент видит только cwd и запись наружу молча отклонит — external_directory):
- `.lock` — дата + id сессии на старте; свежий lock той же цели блокирует параллельный прогон; удали в конце.
- `r1-<роль>.md` — вердикт каждого судьи СРАЗУ по получении.
- `run_state.md` — счётчик вызовов, назначение «роль↔модель» по раундам, текущий этап.
- `verdict.md` — итоговая таблица (6 критериев × 3 судьи + столбец majority) · **minority report**
  (каждое несовпавшее мнение с ⚠️, не отбрасывать: диссентер часто прав в детали) · **traceability
  footer** (роль↔модель ПО КАЖДОМУ раунду, счёт голосов, дата, что судили + версия артефакта).
- **GATE-блок в хвосте КАЖДОГО артефакта** (машиночитаемый): `phase / calls_used / checklist_out /
  next_step / violations`. Исполнитель следующего шага ПЕРВЫМ действием читает GATE предыдущего
  (halt-on-empty); поле пусто/плейсхолдер → 1 целевой retry поля → degraded-start с записью в
  `violations` (без deadlock). Счётчик вызовов живёт в GATE, не «в уме».
Обрыв/компакция сессии между раундами → прогон восстанавливается по файлам и последнему валидному
GATE, а не начинается с нуля.

## Definition of Done
- [ ] Пул ≥3 моделей ≥3 семейств (или явная пометка «пул N/3, кто выпал и почему»).
- [ ] Каждый вердикт — отдельным файлом; в шапке роль, модель, дата, раунд.
- [ ] Итог = majority по каждому критерию + взвешенная сумма; счёт голосов приведён.
- [ ] Все majority-FAIL закрыты правками или задача эскалирована человеку.
- [ ] minority report и traceability footer заполнены; циклов ≤ лимита.
