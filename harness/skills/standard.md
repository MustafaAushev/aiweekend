# skill: standard (dev-скил, язык-агностик playbook)

> Агент читает как playbook «привести код к стандарту разработки». Плейсхолдеры <ТИП-ЧЕКЕР>/<ЛИНТЕР>/<ФОРМАТТЕР>/<ЛОГГЕР>/<CONFIG_LIB>/<ПАКЕТ_МЕНЕДЖЕР>/<VCS> под свой стек. Правила — в agents/*.md (структура, типы, ошибки, логи, конфиг, зависимости, vcs). Пометки П1–П6 — сквозные принципы курса.

## Скил: STANDARD — единый dev-стандарт репозитория (структура · типы · ошибки · логи · конфиг · зависимости · VCS)

**Когда:** любая задача, где агент создаёт или меняет код (feature/bugfix/hotfix/refactor) — стандарт подключается сквозной полосой, как tests/review. Отдельно вызывается на промтах «приведи к стандарту / настрой новый сервис / наведи порядок в проекте / добавь конфиг/логирование/типизацию». Стандарт задаёт КАК устроен production-код между задачами; feature/review проверяют ЧТО делает конкретный diff.

**Preflight:** rules-файлы `agents/{structure,types,errors,logging,config,deps,vcs}.md` и `thresholds.yaml` создаются в блоке harness (часть 2). Проверь: `ls ~/workshops/aiweekend/harness/agents/`. Нет каталога — сначала блок harness, потом сюда.

**Глоссарий (жаргон плейбука):** dev-стандарт: договорённый минимум того, как выглядит и устроен код (структура, типы, ошибки, логи, конфиг, зависимости), не зависящий от конкретной фичи. публичный контракт: то, что видно снаружи модуля/сервиса — сигнатуры экспортируемых функций, схемы сериализации, поля ответа. lock-файл: фиксация точных версий всех (в т.ч. транзитивных) зависимостей + хэши для воспроизводимости и защиты от supply-chain. structured logging: лог как набор полей (event + key=value), а не отформатированная строка — чтобы агрегировать по шаблону. correlation-id: сквозной идентификатор запроса/операции, связывающий логи одной цепочки. fail-fast config: конфиг валидируется при старте — сервис падает сразу, а не через час на первом обращении. layer import matrix: правила, какой слой какой имеет право импортировать (домен ничего проектного, api не лезет в БД напрямую).

**Форма правила в harness:** связать `harness/skills/standard.md` с `agents/structure.md`, `agents/types.md`, `agents/errors.md`, `agents/logging.md`, `agents/config.md`, `agents/deps.md`, `agents/vcs.md`. В эти файлы внести запрещающие правила:

```markdown
## [STANDARD] Dev-стандарт репозитория (нарушение золотого правила = блок PR)
1. Структура: код разложен по слоям/фичам (feature = модель+сервис+репозиторий+ошибки), общий код в shared, никаких «свалок» utils с несвязанными функциями; файл > ~300–400 строк — сигнал к декомпозиции. Матрица импортов соблюдена: домен не импортирует инфраструктуру; api/handler не ходит в БД мимо сервиса; направление зависимостей — внутрь к домену.
2. Типы: все публичные функции/методы аннотированы; <ТИП-ЧЕКЕР> в strict; публичный возврат — типизированный контракт (struct/DTO/schema), НЕ «голый словарь/any»; подавление типов — только с комментарием-причиной.
3. Ошибки: запрещён голый catch-all с проглатыванием; доменная иерархия исключений; try минимальный; технический сбой не маскируется пустым/None-возвратом — пробрасывается с контекстом (wrap-and-reraise).
4. Логи: структурные (event + поля), НЕ интерполяция строк; уровни по критериям (debug/info/warning/error/critical); без секретов/PII (рекурсивная редакция); correlation/request/trace-id через контекст; в prod — машинный формат (JSON), в dev — человекочитаемый; не логировать в горячих циклах.
5. Конфиг/секреты: единый типизированный источник настроек, валидируемый на старте (fail-fast); секреты только из env/vault, НИКОГДА в коде и <VCS>; файл секретов в ignore, в репо — только `.env.example` с плейсхолдерами; приоритет env > файл > дефолты; никаких разбросанных getenv по коду.
6. Зависимости/lock: в манифесте — совместимые диапазоны, в lock — точные версии + хэши; lock закоммичен; CVE-аудит (<CVE_AUDIT>) в CI — красный при известной уязвимости; обновление раз в 2–4 недели; установка без lock запрещена.
7. Формат/линт: авто-форматтер (<ФОРМАТТЕР>) и линтер (<ЛИНТЕР>, включая security-набор) — не обсуждаются; enforced в pre-commit и CI; обход хуков запрещён.
8. VCS: ветки `<type>/<TICKET>-<slug>`, Conventional Commits с тегом вклада `[agent|assisted|manual]`, атомарные коммиты, PR ≤ порога с шаблоном и ≥1 аппрувом, `<DEFAULT_BRANCH>` protected.
9. Документация: публичный API документирован (назначение/аргументы/возврат/ошибки); комментарий объясняет «почему», не «что»; README — обязательные секции (Overview/Quickstart/Config(ENV)/Running/Architecture/API); значимое техрешение → ADR.
```

**Шаги (что делает агент):**

1. **Определить целевой стандарт стека.** Прочитать `agents/{structure,types,errors,logging,config,deps,vcs}.md` и профиль языка (идиома в §11.1). Зафиксировать инструменты: `<ФОРМАТТЕР>`, `<ЛИНТЕР>`, `<ТИП-ЧЕКЕР>`, `<ТЕСТ_РАННЕР>`, `<ЛОГГЕР>`, `<CONFIG_LIB>`, `<ПАКЕТ_МЕНЕДЖЕР>`, `<LOCK_CMD>`.
2. **Структура и границы слоёв.** Разложить/проверить раскладку: слой/фича = модель+сервис+репозиторий+ошибки, общий код в shared, конфиг в одном месте. Проверить матрицу импортов (домен не тянет инфраструктуру; api не ходит в БД мимо сервиса). Файлы > ~300–400 строк — декомпозировать.
3. **Строгая типизация публичного контракта.** Аннотировать все публичные сигнатуры; `<ТИП-ЧЕКЕР>` strict — ноль ошибок. Заменить «голый словарь/any» на границе модуля типизированным контрактом (struct/DTO/schema). Подавления типов — только с комментарием-причиной.
4. **Обработка ошибок.** Убрать голый catch-all и проглатывание; ввести/использовать доменную иерархию исключений; сузить try до опасных строк; технический сбой пробрасывать с контекстом, не подменять пустым/None.
5. **Структурное логирование.** Перевести логи на `<ЛОГГЕР>` с полями (event + key=value), убрать интерполяцию строк и секреты/PII; проставить уровни по критериям; привязать correlation/request-id через контекст; prod — машинный формат, dev — человекочитаемый.
6. **Конфиг и секреты.** Свести настройки в единый типизированный источник (`<CONFIG_LIB>`), валидируемый на старте (fail-fast). Все секреты — из env/vault; вынести хардкод; добавить `.env.example` с плейсхолдерами, `.env`/секреты — в ignore. Убрать разбросанные getenv.
7. **Зависимости и lock.** В манифесте — диапазоны, в lock — точные версии + хэши (`<LOCK_CMD>`); закоммитить lock; прогнать `<CVE_AUDIT>`. Тронул зависимости — пересобрал lock тем же PR.
8. **Формат/линт/хуки.** Прогнать `<ФОРМАТТЕР>` + `<ЛИНТЕР>` (incl. security-набор) + `<ТИП-ЧЕКЕР>`; настроить pre-commit; не обходить хуки.
9. **Документация и ADR.** Документировать публичный API; привести README к обязательным секциям; значимое техрешение — ADR, а не комментарий.
10. **VCS-гигиена.** Работа в ветке `<type>/<TICKET>-<slug>`; коммиты Conventional + тег вклада `[agent|assisted|manual]`; PR по шаблону, ≤ порога, ≥1 аппрув, merge только на зелёном CI. Значимое отступление от стандарта — через ADR, не молча.

**Definition of Done:**

- [ ] Структура: раскладка по слоям/фичам, общий код в shared, нет «свалок»; матрица импортов не нарушена; крупные файлы декомпозированы.
- [ ] Типы: публичные сигнатуры аннотированы, `<ТИП-ЧЕКЕР>` strict зелёный; публичный возврат — типизированный контракт, не «голый словарь/any»; подавления прокомментированы.
- [ ] Ошибки: нет голого catch-all/проглатывания; доменная иерархия; технический сбой не замаскирован пустым/None.
- [ ] Логи: структурные, без секретов/PII, с correlation-id, уровни по критериям, prod-формат машинный.
- [ ] Конфиг: единый типизированный источник, fail-fast на старте; секреты из env/vault; `.env` в ignore, `.env.example` в репо; getenv не разбросан.
- [ ] Зависимости: манифест-диапазоны + lock с хэшами закоммичены; `<CVE_AUDIT>` зелёный.
- [ ] Формат/линт/тип-чек проходят; pre-commit настроен; хуки не обойдены.
- [ ] Публичный API документирован; README с обязательными секциями; значимое решение → ADR.
- [ ] VCS: ветка `<type>/<TICKET>-<slug>`, Conventional + тег вклада, PR по шаблону/≤порога/≥1 аппрув; `<DEFAULT_BRANCH>` не тронут напрямую.

**Eval (как harness проверяет):**

> Команды ниже — референс-гейты `make eval-standard` под свой стек, не готовый скрипт воркшопа: инструменты и пороги задаёшь сам (`thresholds.yaml`). Где взять: линтер+формат (`<ЛИНТЕР>`/`<ФОРМАТТЕР>`), тип-чекер (`<ТИП-ЧЕКЕР>`), CVE-аудит (`<CVE_AUDIT>`: pip-audit/govulncheck/npm audit), secret-scan (`<SECRET_SCAN>`: git-secrets/truffleHog/gitleaks), SAST (`<SAST>`: Semgrep/bandit-набор).

CI-гейт `make eval-standard` (bash/Makefile) обязателен до merge:
1. `format-gate`: `<ФОРМАТТЕР> --check` + `<ЛИНТЕР>` (incl. security-набор) — 0 ошибок.
2. `type-gate`: `<ТИП-ЧЕКЕР>` в strict — 0 ошибок; отдельно — grep/AST на «голый словарь/any» в публичном возврате → fail.
3. `error-gate`: статик-чек на голый catch-all и проглатывание исключений (пустой обработчик) → fail.
4. `logging-gate`: нет интерполяции строк в вызовах логгера, нет секретов/PII (по словарю `password|token|secret|api_key|authorization|cookie`); лог-строки содержат correlation-id на request-путях.
5. `config-gate`: `<SECRET_SCAN>` — 0 хардкод-секретов; конфиг грузится через единый типизированный источник; сервис падает при отсутствии обязательной переменной (fail-fast smoke).
6. `deps-gate`: lock присутствует и с хэшами; `<CVE_AUDIT>` (strict) зелёный; манифест и lock согласованы.
7. `docs-gate`: публичные символы имеют docstring/док-комментарий; README содержит обязательные секции.
8. `vcs-gate`: сообщения матчат `^\[(agent|assisted|manual)\] (feat|fix|refactor|perf|test|docs|build|ci|chore)(\(.+\))?: `; ветка ≠ `<DEFAULT_BRANCH>`; PR ≤ `vcs.pr_max_lines`.
9. Бюджет gate: eval ≤ 10 мин CPU; пороги строгости (какой пункт блок, какой предупреждение) — в `thresholds.yaml quality.standard_*`.

**Per-язык идиома:**

- **Go:** раскладка `internal/<feature>/{model,service,repo}` + `cmd/`; типы — компилятор + `golangci-lint` (`errcheck`, `gosec`, `govet`); ошибки — `fmt.Errorf("...: %w", err)` (wrap), sentinel/typed errors, `errors.Is/As`; логи — `slog`/`zap` (`slog.Info("event", "key", val)`); конфиг — `envconfig`/`viper` + валидация на старте; lock — `go.mod`/`go.sum` (хэши встроены), CVE — `govulncheck`; VCS — Conventional Commits + `commitlint`.
- **`<ВАШ_СТЕК>`:** формат/линт (incl. security) + строгий тип-чекер + типизированный контракт на границе + доменные ошибки с wrap + структурный логгер с correlation-id + типизированный конфиг с fail-fast + lock с хэшами + CVE-аудит + Conventional-коммиты с тегом вклада (подставь идиомы своего стека или удали строку из профиля).
- _(конкретику под язык — `<ФОРМАТТЕР>`/`<ЛИНТЕР>`/`<ТИП-ЧЕКЕР>`/`<ЛОГГЕР>`/`<CONFIG_LIB>`/`<LOCK_CMD>`/`<CVE_AUDIT>` — материализует совет; профиль-пример Python в §11.1)_

🎬 **Что ломается без скила:**

Агент собирает «код из туториала»: возвращает из публичного метода нетипизированный словарь (клиент угадывает поля, тип-чекер молчит), глушит ошибки голым catch-all и отдаёт None вместо исключения (сбой БД выглядит как «данных нет»), пишет `log("user " + id + " created")` — агрегировать и искать по логам невозможно, а рядом в строку попадает токен. Секреты — хардкодом в коде и в истории <VCS>; конфиг читается разбросанными getenv без валидации, поэтому сервис стартует и падает через час на первом обращении к отсутствующей переменной. Зависимости без lock и хэшей — сборка невоспроизводима, свежая CVE въезжает незамеченной. Слои перепутаны (handler ходит в БД мимо сервиса), файл на 900 строк, публичного API-документации нет. Каждый следующий diff агента усиливает энтропию: при том же промте он воспроизводит те же дефекты, ревью тонет в мелочах вместо логики, а онбординг нового человека (или агента) занимает дни вместо часов.
