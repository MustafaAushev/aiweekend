# skill: database (dev-скил, язык-агностик playbook)

> Агент читает как playbook задачи. Плейсхолдеры <БД>/<ORM>/<ДРАЙВЕР>/<МИГРАЦИИ>/<ТЕСТ_РАННЕР> под свой стек. Правила — в agents/*.md.

## Скил: DATABASE — работа с базой данных агентом

**Когда:** Задача трогает хранилище — «добавь таблицу/поле», «напиши запрос/репозиторий», «сделай миграцию», «оптимизируй медленный запрос», «выгрузи/массово обнови данные»; любой стек (<БД> = PostgreSQL/MySQL/ClickHouse/…, любой <ORM>/<ДРАЙВЕР>); агенту разрешена только реализация — схема, миграции и прод-доступ под защитой человека.

**Preflight (предполётная проверка окружения):** файлы `agents/*.md`, `thresholds.yaml`, каталог миграций `<МИГРАЦИИ>/` и эфемерная тестовая <БД> (testcontainers / одноразовый инстанс) создаются в блоке harness, а не в этом плейбуке. Перед запуском скила: `ls harness/agents/` и `ls <МИГРАЦИИ>/`. Секреты подключения — только из env (`DATABASE_URL`/secret-manager), никогда в коде, логах или промпте. Каталога миграций нет — сначала подними инфраструктуру миграций, потом возвращайся сюда.

**Глоссарий (жаргон плейбука):** expand-contract: миграция в две фазы — сначала расширяем (добавляем совместимое), деплоим код, потом сужаем (убираем старое). forward-only: на проде только вперёд, откат — новой миграцией-компенсацией, не `down` живой таблицы. zero-downtime: схема меняется без остановки, старый и новый код работают на одной БД одновременно. N+1: 1 запрос на список + по запросу на каждый элемент вместо одного batch/JOIN. keyset (seek) pagination: страница по `WHERE key > :cursor ORDER BY key LIMIT n`, а не `OFFSET`. идемпотентность записи: повтор мутации не создаёт дубль (natural/idempotency-key + UNIQUE/UPSERT). оптимистичная блокировка: version/`updated_at` в `WHERE`, конфликт → ретрай. пессимистичная: `SELECT … FOR UPDATE`. уровень изоляции: RC/RR/Serializable — что видит транзакция и от каких аномалий защищена. deadlock: две транзакции крест-накрест ждут замки друг друга. lost update / потерянный commit: два апдейта затёрли друг друга. EXPLAIN (ANALYZE): план запроса (и факт с таймингом и прочитанными строками). connection pool: пул переиспользуемых соединений с лимитом. least-privilege: минимально необходимые GRANT'ы роли.

**Форма правила в harness:** Дописать в `harness/skills/database.md` и связать с `agents/contracts.md`, `agents/migrations.md`, `agents/performance.md`, `agents/resilience.md`, `agents/security.md`, `agents/observability.md`, `agents/test-matrix.md` (и `agents/incident-log.md` — не по умолчанию, только если всплыл новый класс отказа: deadlock-шторм, потерянный commit, миграция без отката, full-scan на горячем пути):

```markdown
## [DATABASE] Доступ к данным
Агенту запрещено писать код доступа к данным, пока:
1. Нет изменения схемы как версионированной обратимой миграции в отдельном файле (<МИГРАЦИИ>), с проверенным откатом на тестовой БД; на проде — forward-only, zero-downtime (expand-contract).
2. Нет объявленных инвариантов целостности в самой схеме: NOT NULL, UNIQUE, FK (ON DELETE), CHECK — а не только «в коде проверю».
3. Нет failing-теста на новое поведение репозитория/запроса против реальной <БД> (testcontainers/ephemeral, не мок <ДРАЙВЕР>).
4. Нет плана нагрузки на запрос: ожидаемый объём строк, целевой p99 и индекс, который его держит (подтверждён EXPLAIN).
Запрещено на горячем пути: full-scan (Seq Scan по большой таблице), N+1, OFFSET-пагинация на растущей таблице, SELECT * без нужды, конкатенация значений в SQL.
Обязательно: parameterized/prepared запросы; транзакция с явным уровнем изоляции на мультистейтмент-записи; идемпотентность мутаций (idempotency-key + UNIQUE/UPSERT); statement_timeout на каждый запрос + retry только на deadlock/serialization (≤3, backoff+jitter); connection pool с лимитом и таймаутом ожидания; least-privilege роль; секреты из env; correlation_id в теге/комментарии запроса.
Каждый коммит агента: миграция (up+down) + тест против реальной БД + EXPLAIN горячего запроса.
```

**Шаги (что делает агент):**

1. **Схема первой (контракт данных).** Прочитать текущую схему (`\d` / `SHOW CREATE TABLE` / каталог). Изменение оформить как diff схемы: таблицы, типы, NOT NULL, DEFAULT, UNIQUE, FK с `ON DELETE/UPDATE`, CHECK-инварианты, индексы. Инварианты держать в БД (constraints), а не только в приложении: БД — последняя линия целостности, приложение можно обойти. Не трогать прод-схему напрямую, только через миграцию.
2. **Версионированная обратимая миграция.** Каждое изменение схемы — отдельный файл в `<МИГРАЦИИ>` (alembic/Flyway/Liquibase/golang-migrate/…): `up` + `down`. Прогнать up→down→up на тестовой БД — откат обязан возвращать схему байт-в-байт. На проде — forward-only и zero-downtime по expand-contract: (a) добавить новое nullable/с DEFAULT, задеплоить код, читающий оба варианта; (b) бэкфилл батчами; (c) переключить запись; (d) отдельной поздней миграцией удалить старое. Никаких `DROP COLUMN` / `SET NOT NULL` / переименований в одну фазу на живой таблице — это ломает работающий код на выкатке.
3. **Failing test раньше кода.** Написать тест против реальной <БД> (testcontainers/ephemeral instance, не мок <ДРАЙВЕР>): контракт репозитория, инварианты (нарушение UNIQUE/FK/CHECK ⇒ ошибка БД), поведение конкурентных транзакций. Зелено без реализации ⇒ тест ничего не требует, работа остановлена.
4. **Запросы: параметризованные и под план.** Только parameterized/prepared — значения передаются отдельно от текста SQL (анти-инъекция), никакой конкатенации/format-строк. Для каждого запроса горячего пути — `EXPLAIN` (и `EXPLAIN ANALYZE` / `EXPLAIN ESTIMATE` на копии данных): убедиться, что идёт по индексу, а не Seq/full scan, и что прочитанные строки в пределах бюджета из `thresholds.yaml`. Нет индекса под предикат/сортировку/JOIN-ключ — добавить его миграцией (шаг 2), а не запускать вслепую.
5. **N+1 → batch/eager.** Список сущностей + связанные — одним запросом (JOIN / `IN (…)` / dataloader / `<ORM>`-eager/prefetch), не циклом «запрос на элемент». В тесте — счётчик SQL-запросов: на N сущностей число запросов постоянно, а не растёт с N.
6. **Пагинация keyset, не OFFSET.** Списки — seek-пагинация: `WHERE (sort_key) > :cursor ORDER BY sort_key LIMIT :n`. `OFFSET N` на растущей таблице читает и выбрасывает N строк на каждый листинг — деградирует линейно и уходит в full-scan на глубоких страницах. OFFSET допустим только на заведомо маленьком/ограниченном сверху наборе.
7. **Транзакции, изоляция, блокировки.** Мультистейтмент-запись — в одной транзакции с ЯВНЫМ уровнем изоляции (RC по умолчанию; RR/Serializable там, где нужна защита от phantom/write-skew). Конкурентная правка одной строки: оптимистичная блокировка (version/`updated_at` в `WHERE`, 0 обновлённых строк ⇒ конфликт ⇒ ретрай) или пессимистичная (`SELECT … FOR UPDATE`) — выбор обосновать нагрузкой. Транзакции короткие; единый порядок захвата ресурсов (профилактика deadlock); НИКАКИХ сетевых/LLM/HTTP-вызовов внутри открытой транзакции — они держат замки и рвут пул.
8. **Идемпотентность записи.** Мутация с внешним эффектом — идемпотентна: natural key или idempotency-key + `UNIQUE`-constraint + `INSERT … ON CONFLICT DO NOTHING/UPDATE` (UPSERT) либо `INSERT … WHERE NOT EXISTS`. Повтор (retry, дубль сообщения из брокера, двойной клик) не создаёт вторую запись и не списывает дважды.
9. **Пул, таймауты, ретраи.** Ходить в БД только через connection pool (лимит соединений ≤ ёмкости <БД>, таймаут ожидания свободного, max lifetime — иначе пул-исчерпание кладёт сервис раньше самой БД). На каждый запрос — `statement_timeout`/query timeout (из `thresholds.yaml`). Ретрай ТОЛЬКО на транзиентном (deadlock 40P01, serialization failure 40001, обрыв соединения): exp backoff + jitter, ≤3; бизнес-ошибки и constraint-violation не ретраить. Ретрай мутации безопасен лишь при идемпотентности (шаг 8), иначе retry-шторм = двойные списания.
10. **Большие объёмы: стриминг и батчи.** Большая выборка — серверный курсор/стриминг с `LIMIT`/чанками, не materialize всего в память приложения. Массовая запись/бэкфилл/удаление — батчами (напр. 1–10k строк, чанки по ключу) с паузами, чтобы не держать длинную транзакцию, не раздувать WAL/undo-лог и не лочить таблицу целиком; на проде — вне пиковых часов.
11. **Безопасность доступа.** Роль приложения — least-privilege (только нужные таблицы/операции; отдельная read-only роль для аналитики/агента; без DDL и без доступа к PII сверх необходимого в рантайме). Секреты подключения — из env/secret-manager, никогда в коде/логах/промпте. Не тащить лишние колонки (особенно PII) в выборки и логи.
12. **Наблюдаемость.** Тег/комментарий каждого запроса с `correlation_id` (`/* app:svc trace:<id> */`) — чтобы связать SQL в slow-query log с запросом приложения. Включить slow-query log и метрики: latency запросов, размер/время ожидания пула, доля ретраев, deadlock-rate, прочитанные строки. Долгий/тяжёлый запрос обязан находиться по трейсу за секунды, а не гриплянием логов вручную.

**Definition of Done:**

- [ ] Изменение схемы — версионированная миграция (up+down), откат проверен на тестовой БД; прод-план zero-downtime (expand-contract) описан.
- [ ] Инварианты в схеме: NOT NULL / UNIQUE / FK(ON DELETE) / CHECK — не только в коде приложения.
- [ ] ≥1 тест против реальной <БД> (testcontainers): контракт репозитория + нарушение хотя бы одного constraint + конкурентная транзакция.
- [ ] Все запросы parameterized/prepared; в диффе нет конкатенации значений в SQL.
- [ ] EXPLAIN горячих запросов приложен: индексный доступ, без Seq/full scan; прочитанные строки ≤ бюджета.
- [ ] Нет N+1 (тест на постоянное число запросов); списки — keyset-пагинация, не OFFSET.
- [ ] Мутации: транзакция с явной изоляцией + идемпотентность (idempotency-key/UNIQUE/UPSERT) + retry на deadlock/serialization.
- [ ] Пул с лимитом + statement_timeout на каждый запрос; ретраи только транзиентные, ≤3, backoff+jitter.
- [ ] Большие выборки — стриминг/курсор; массовые операции — батчами.
- [ ] Least-privilege роль; секреты из env; correlation_id в запросах; slow-query log и метрики пула включены.

**Eval (как harness проверяет):**

> Команды ниже — референс-гейты под свой стек, а не готовый скрипт воркшопа: свой `Makefile`, свою <БД> в testcontainers и SLO задаёшь сам. Expected: зелёный гейт, все проверки проходят до merge. Где взять инструменты: testcontainers (testcontainers.com), pgTAP/sqlc, pgbench (входит в postgres) / k6 (k6.io), sqlfluff + sqlmap (инъекции), git-secrets/truffleHog (секреты).

CI-гейт `make eval-database` обязателен до merge:
1. `migration-gate`: применить все миграции на чистой <БД>, затем `down` последней и снова `up` — схема идентична исходной; drift между схемой и <ORM>/спекой блокирует merge.
2. `constraint-gate`: тесты нарушают UNIQUE/FK/CHECK/NOT NULL — БД обязана отклонить каждую (никаких «молча прошло»).
3. `plan-gate`: `EXPLAIN (ANALYZE, BUFFERS)` горячих запросов на репрезентативном объёме — fail при Seq Scan / full scan по большой таблице или прочитанных строках > бюджета.
4. `nplus1-gate`: перехват SQL в тесте — число запросов на список константно; рост с N ⇒ fail.
5. `concurrency-gate`: две параллельные транзакции на одну строку — оптимистика ловит конфликт и ретраит, deadlock не роняет запрос (fast-fail + retry ≤3), потерянных обновлений нет.
6. `idempotency-gate`: одна и та же мутация дважды (тот же idempotency-key) → ровно одна запись / один эффект.
7. `security-gate`: статический анализ на конкатенацию в SQL + fuzz инъекцией (`' OR 1=1--`) ⇒ параметризация держит; проверка least-privilege (DDL/запись сверх роли отклонены); secret-scan подключения.
8. `load-gate`: pgbench/k6 на 10× данных — p99 ≤ SLO, пул не исчерпывается, deadlock-rate ≈ 0, память/соединения не растут неограниченно.
9. Бюджет gate: eval ≤ N мин CPU; миграция, прогнанная на копии прод-объёма, укладывается в деплой-окно.

🎬 **Что ломается без скила:**

Агент реализует «список заказов с товарами» циклом: 1 запрос на заказы + по запросу на товары каждого — **N+1**; на демо (10 заказов) незаметно, на проде 5000 заказов = 5001 запрос, пул соединений исчерпан, БД в CPU-полке, соседние сервисы ловят таймауты. Пагинацию делает `OFFSET`: на первой странице миллионной таблицы мгновенно, на 900 000-й БД читает и выбрасывает 900k строк на каждый листинг — **OFFSET на миллионах**. Схему меняет напрямую `ALTER TABLE … DROP COLUMN` без миграции и отката — деплой не разворачивается, восстановление руками ночью (**миграция без отката**). Мутацию баланса пишет без транзакции и без idempotency-key — retry клиента списывает дважды, а два параллельных апдейта затирают друг друга (**потерянный commit**, lost update). Значения подставляет конкатенацией в SQL — `'; DROP TABLE …` проходит (**SQL-инъекция**). Запросы без индекса под предикат — **full-scan** горячего пути; без statement_timeout один тяжёлый `GROUP BY` висит и держит соединения. Транзакции длинные, порядок захвата замков разный — под нагрузкой **deadlock-шторм**, БД откатывает пачками, пользователи видят 500. Массовое удаление одной транзакцией на 10 млн строк рвёт WAL и блокирует таблицу. Итог: инцидент без следов — ни correlation_id, ни slow-query log; расследование на дни, откат не помогает (агент при той же спеке/промпте воспроизводит тот же N+1 и ту же нехватку индекса).
